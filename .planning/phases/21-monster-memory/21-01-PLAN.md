# Phase 21 Plan 01 — MonsterMemory class + slimmed-context augmentation

## Objective

Add an in-memory `MonsterMemory` class (bounded LRU, three signals, INT-gated marking) at `src/eldritch_dm/gameplay/monster_memory.py`, plus a `_augment_with_memory` helper and an optional `monster_memory` ctor arg on `SmartMonsterDriver` that augments slimmed candidates with three D-57-safe derived flags before the LLM call. Zero regression in existing tests. No persistence, no cog wiring (Plan 21-02 covers persistence; cog wiring is documented as a v1.7 follow-up).

## Locked Implementation Decisions (resolve CONTEXT ambiguities)

| Tag | Decision | Rationale |
|-----|----------|-----------|
| **L-01** | `damage_dealt_by` is **cumulative-this-session**, not windowed. The LLM-facing field is named `recent_damage_dealt` (per D-159) but is computed from the cumulative total. "Recent" semantically reflects "within this session," which is bounded by `purge_session`. | Matches D-158 storage shape; avoids round-windowed bookkeeping; in practice combats rarely span >5 rounds so cumulative IS recent. |
| **L-02** | Damage bands (D-159 boundaries made explicit): `low` = `< 5`, `moderate` = `5..14` inclusive, `high` = `>= 15`. Defined as module constants `DAMAGE_BAND_MODERATE_MIN = 5`, `DAMAGE_BAND_HIGH_MIN = 15`. | D-159 left boundaries fuzzy; lock them so tests are deterministic. |
| **L-03** | "200 round-events per session" is implemented as a bounded `collections.deque(maxlen=200)` of raw `(round_number, kind, pc_id, value)` events, kept alongside the signal dicts. Signal dicts (`damage_dealt_by`, `concentrating_on`, `marked_dangerous`) are unbounded but party-sized (~6 PCs) so growth is negligible. Deque eviction is purely for audit/replay and does NOT roll back signal totals — the totals stay correct. | Lightest implementation that satisfies D-157 LRU intent. |
| **L-04** | `_slim_candidate` signature is **frozen** — it is imported by name in `tests/gameplay/test_smart_monster_driver.py:30` and 96+ tests pivot on it. New behavior is added via a separate module-level `_augment_with_memory(slim, memory, pc_id, monster_int)` helper called in `_pick_target_llm` AFTER `_slim_candidate`. | Zero regression; clean unit-test seam. |
| **L-05** | `SmartMonsterDriver` gains optional `monster_memory: MonsterMemory \| None = None` ctor arg. Default `None` → augmentation skipped → byte-identical to current behavior. Factory default also `None`. | Zero-regression guarantee. |
| **L-06** | INT-gated marking: `marked_dangerous` is set ONLY when the observing monster's INT ≥ 10. Below 10 the set never grows for that monster. The monster's INT is captured at `observe_hit` time (passed by the caller) — MonsterMemory does not look it up. | Mirrors v1.1 D-53 INT-gating philosophy; keeps MonsterMemory free of monster-stat lookups. |
| **L-07** | Fail-soft (D-165) applied at three boundaries: (a) `MonsterMemory.observe_*` swallows exceptions and logs; (b) `recall_memory` returns an empty `MonsterMemory()` on lookup error; (c) `_augment_with_memory` returns the unaugmented slim dict if any error occurs. None of these can crash combat. | D-58/D-165 contract. |
| **L-08** | `concentrating_on` is per-`pc_id`. When a PC ends/breaks concentration, the caller is expected to call `observe_concentration(pc_id, spell=None)` to clear. Until then the value sticks. This Plan does NOT auto-expire concentration — wiring concentration-broken events is part of the v1.7 follow-up gap. | Keeps MonsterMemory passive; bot/rules-engine owns liveness. |

## Tasks

<task id="1" type="auto">
  <name>Add MonsterMemory class</name>
  <behavior>
    `src/eldritch_dm/gameplay/monster_memory.py` exposes a `MonsterMemory` dataclass-like class
    with:
    - state: `damage_dealt_by: dict[str, int]`, `concentrating_on: dict[str, str | None]`,
      `marked_dangerous: set[str]`, plus internal `_events: collections.deque[tuple]` (maxlen=200).
    - `observe_hit(pc_id: str, damage: int, *, round_number: int = 0, observer_int: int | None = None) -> None`:
        adds to `damage_dealt_by`, appends event, sets `marked_dangerous` IFF `observer_int is not None and observer_int >= 10`.
    - `observe_concentration(pc_id: str, spell: str | None, *, round_number: int = 0) -> None`:
        sets/clears `concentrating_on[pc_id]`.
    - `damage_band(pc_id: str) -> Literal["low","moderate","high"] | None`: returns `None` if pc unseen, else band per L-02.
    - `snapshot_dict() -> dict`: pydantic-serializable shape for Plan 21-02 persistence (keys: `damage_dealt_by`, `concentrating_on`, `marked_dangerous`, no event log).
    - `from_snapshot(d: dict) -> MonsterMemory` classmethod: inverse.
    - All `observe_*` wrapped in `try/except` (fail-soft per L-07); logs via `structlog.get_logger().bind(component="monster_memory")`.
    NEVER store HP or AC. NEVER store exact damage in the LLM-facing surface (kept only in `damage_dealt_by` for the band computation).
  </behavior>

<task id="2" type="auto">
  <name>Add MonsterMemoryRegistry helper</name>
  <behavior>
    Same `monster_memory.py` module exposes `MonsterMemoryRegistry` — a tiny in-process dict keyed by
    `(channel_id, session_id, monster_id) -> MonsterMemory`. Methods:
    - `recall(channel_id, session_id, monster_id) -> MonsterMemory`: get-or-create; never raises.
    - `purge_session(channel_id, session_id) -> int`: drops all entries matching the prefix; returns
      count purged. Used by Plan 21-02's session-close hook (when wired).
    - `clear()` for tests.
    Registry holds NO persistence — it's the in-memory cache. Plan 21-02 adds an optional repo
    that hydrates/flushes the registry.
  </behavior>

<task id="3" type="auto">
  <name>Add _augment_with_memory helper and integrate into _pick_target_llm</name>
  <behavior>
    In `src/eldritch_dm/gameplay/smart_monster_driver.py`:
    - Add module-level `_augment_with_memory(slim: dict, memory: MonsterMemory | None, *, pc_id: str) -> dict`:
      Returns `slim` unchanged if `memory is None`. Otherwise returns a NEW dict (does not mutate input)
      adding three keys:
        - `recent_damage_dealt`: result of `memory.damage_band(pc_id)` (omitted/None if unseen — pass through)
        - `concentrating_on`: `memory.concentrating_on.get(pc_id)`
        - `marked_dangerous`: `pc_id in memory.marked_dangerous`
      Wrapped in try/except → returns `slim` on any failure (L-07).
    - `SmartMonsterDriver.__init__` gains `monster_memory: MonsterMemoryRegistry | None = None` kwarg
      stored as `self._monster_memory`. Default `None`.
    - In `_pick_target_llm`, AFTER `candidates = [_slim_candidate(p) for p in targets]`, if
      `self._monster_memory is not None`, look up via
      `mem = self._monster_memory.recall(channel_id, session_id_or_"unknown", monster_id)` and rebuild
      candidates as `[_augment_with_memory(c, mem, pc_id=c["id"]) for c in candidates]`.
      (Note: `session_id` is not currently a parameter of `_pick_target_llm` — we fall back to the
      string `"default"` since the registry key just needs a stable string. A future plan may thread
      real session_id through.)

      already serialized as part of the JSON candidate list (no prompt-template change needed because
      candidates are dumped via `json.dumps`).
    - Confirm NEVER pass `hp_current`, `hp_max`, `ac` raw numerical damage, or class/subclass — those
      stay exactly as `_slim_candidate` already permits.
