# Phase 12 Plan 02 — 50-scenario corpus + eldritch-dm-eval CLI + aggregator/reporter

## Objective

Ship the eval flywheel end-to-end:

1. `tests/eval/dataset/tactical_corpus.jsonl` — 50 hand-authored scenarios
   across 5 archetypes (10 each), all original Apache-2.0 content. License
   header on line 1 (JSON object with `license` key — JSONL parsers will
   tolerate this since the loader can skip leading comment-style lines OR

   driver → judge → aggregator → reporter → JSON + Markdown outputs.
5. New `[project.scripts]` entry `eldritch-dm-eval`.
6. Tests: aggregator math, baseline-diff exit codes, end-to-end CLI smoke
   with mocked LLM and 1-scenario subset.

## Critical sub-decisions

### S-12-02-A — Driver invocation bypass strategy
The CLI needs to invoke `SmartMonsterDriver._choose_target` directly, without
the surrounding combat orchestration. The simplest path: construct a minimal
SmartMonsterDriver with stub no-op dependencies for everything EXCEPT the
LLM client + (optionally) the eligibility set + the random_choice fn. We
do NOT need `mcp`, `pc_classes_repo`, `riposte_timers_repo`, `button_factory`,
`state_provider`, or `channel_resolver` — `_choose_target` only uses
`self._openai_client`, `self._llm_model`, `self._llm_timeout_seconds`,
`self._cache`, `self._random_choice`, `self._log`, and the route helper.
The CLI builds a stripped-down driver via a new helper
`eval.runner.build_eval_driver(*, openai_client, model)` that passes `None`
or `_Sentinel()` for the unused deps. Document in the runner's docstring
that this is an EVAL-ONLY construction path; production must continue using
the factory.

### S-12-02-B — Exit-code precedence
D-79 must be checked in this order:
1. **Critical first**: any `per_dimension` average across all scenarios < 0.5 → exit 2.
2. **Regression second**: baseline supplied AND `(baseline.overall_mean - current.overall_mean) > 0.05` → exit 1.
3. **Pass last**: overall mean ≥ 0.7 AND no critical → exit 0.

A run that is BOTH critical AND regressed exits 2 (critical wins). This is
explicit because the failure modes have different operator semantics —
critical means "ship blocker", regression means "investigate before tagging".
Tested in `test_aggregator.py::test_exit_code_critical_beats_regression`.

### S-12-02-C — Corpus license header strategy
JSONL purity matters for diff-ability in PRs. We put the Apache-2.0 + corpus
provenance in `tests/eval/dataset/LICENSE.md` alongside the .jsonl file, and
the loader (from Plan 01) treats every non-empty line as a ScenarioEntry.
The CLI's `--help` output mentions both files for clarity.

### S-12-02-D — Output filename + reproducibility
`eval-{ISO timestamp}-{git short sha}.json` and `eval-{...}.md`. Git short
SHA captured via `subprocess.run(["git", "rev-parse", "--short", "HEAD"], ...)`
with fallback to `"unknown"` when not a git checkout. JSON output includes:
`{"judge_prompt_version", "driver_model", "judge_model", "scenarios": [...],
"aggregate": {...}, "baseline_diff": {...|null}, "started_at", "finished_at"}`.

### S-12-02-E — Mocked-LLM smoke test
The end-to-end CLI smoke test uses `monkeypatch` to replace the AsyncOpenAI
client factory and a 1-scenario JSONL fixture. It DOES NOT spawn a real
oMLX server. Real-LLM eval is operator-run; CI runs the mocked smoke only.

## Tasks

### T-12-02-01 — Aggregator (no baseline diff yet)

**Files:**
- `src/eldritch_dm/eval/aggregator.py` (new)
- `tests/eval/test_aggregator.py` (new)

**Behavior:**
- `ScenarioResult` pydantic model: `{scenario_id, archetype, driver_target_pc_id: str | None, driver_rationale: str | None, verdict: JudgeVerdict | None, judge_error: str | None, latency_ms_driver: int, latency_ms_judge: int}`.
- `AggregateStats` pydantic model: `{n: int, overall_mean: float, per_dimension_mean: dict[str, float], per_archetype_mean: dict[str, float], judge_failure_count: int, driver_failure_count: int}`.
- `aggregate(results: list[ScenarioResult]) -> AggregateStats`:
  - Treats None verdicts as 0.0 across all dimensions (S-12-01-C).
  - Computes overall mean = mean of all per-scenario overall_scores (with None → 0.0).
  - Per-dimension mean: mean of each dimension across all scenarios.
  - Per-archetype mean: mean of overall scores grouped by archetype.

**Tests:**
- Synthetic 5 results, all 1.0 → `overall_mean = 1.0`.
- Synthetic mixed: 4 passes (overall=0.8), 1 fail (None) → `overall_mean = (0.8*4 + 0)/5 = 0.64`.
- Per-archetype grouping: 2 brutes (1.0, 0.5) + 1 spellcaster (0.8) → `{"brute": 0.75, "spellcaster": 0.8}`.
- Per-dimension means computed independently.
- Judge failure count tallied separately from driver failure count.

**Verification:** `pytest tests/eval/test_aggregator.py::test_basic -v` passes.

**Commit:** `feat(12-02): aggregator with per-archetype + per-dimension stats`

### T-12-02-02 — Baseline diff + exit-code derivation

**Files:**
- `src/eldritch_dm/eval/aggregator.py` (extend)
- `tests/eval/test_aggregator.py` (extend)

**Behavior:**
- `BaselineDiff` pydantic model: `{baseline_overall_mean: float, current_overall_mean: float, delta: float, regressed: bool, regression_threshold: float = 0.05}`.
- `compute_baseline_diff(current: AggregateStats, baseline_path: Path) -> BaselineDiff` — reads prior JSON, extracts `aggregate.overall_mean`, computes delta.
- `derive_exit_code(stats: AggregateStats, baseline_diff: BaselineDiff | None) -> int` — implements S-12-02-B precedence: critical (2) > regression (1) > pass (0).

**Tests:**
- `test_exit_code_pass`: overall=0.85, all dims ≥ 0.6 → 0.
- `test_exit_code_critical_any_dim_below_0_5`: overall=0.8 BUT one dim_mean=0.4 → 2.
- `test_exit_code_regression`: overall=0.65, no critical, baseline=0.75, delta=-0.10 → 1.
- `test_exit_code_critical_beats_regression`: overall=0.5, one dim=0.4, baseline=0.85 → 2 (not 1).
- `test_exit_code_no_baseline_no_regression`: overall=0.65, no baseline supplied, no critical → 0 (passes the ≥0.7 check? — design: ≥0.7 threshold is the pass bar; 0.65 = exit 1 even without baseline? Re-read D-79: "0 = passed (avg overall ≥ 0.7, no dimension avg < 0.5)" — so 0.65 with no baseline and no critical is NOT a pass. Decision: when no baseline AND below 0.7 → exit 1 (treat as regression-from-implicit-baseline-of-0.7). Document this.)

**Verification:** `pytest tests/eval/test_aggregator.py -v` passes.

**Commit:** `feat(12-02): baseline diff + exit-code precedence (critical > regression > pass)`

### T-12-02-04 — Eval runner (driver invocation bypassing Discord)

**Files:**
- `src/eldritch_dm/eval/runner.py` (new)
- `tests/eval/test_runner.py` (new)

**Behavior:**
- `build_eval_driver(*, openai_client: AsyncOpenAI, model: str, llm_timeout_seconds: float = 1.5) -> SmartMonsterDriver` — constructs a stripped-down driver suitable for `_choose_target` calls only. Stubs out `mcp`, repos, button_factory, state_provider, channel_resolver with `_NoopSentinel()` objects that raise on access (so any accidental orchestrator-path execution fails loud).
- `async run_scenario(driver, judge, scenario: ScenarioEntry, *, driver_model: str, judge_model: str) -> ScenarioResult`:
  1. Construct `targets` list from `scenario.pc_list` (dict form expected by `_choose_target`).
  2. Construct `current_actor` dict from `scenario.monster_stats` (with the intelligence + character_id).
  3. Call `await driver._choose_target(targets, channel_id=f"eval-{scenario_id}", round_number=1, current_actor=current_actor)` with timing.
  4. Extract `target_pc_id` from the returned PC dict. Note: SmartMonsterDriver's `_pick_target_llm` populates `self._cache` keyed on `(channel, round, monster_id)`; we use a unique channel per scenario so no cross-scenario contamination.
  5. Call `await judge.score(scenario, driver_choice_pc_id=target_pc_id, driver_rationale=None, driver_model=driver_model)` with timing.
  6. Return `ScenarioResult(...)`.
- Driver rationale: SmartMonsterDriver currently does not surface the rationale to the caller (the inner choice has a `rationale` field but it's not returned by `_choose_target`). Decision: pass `driver_rationale=None` for v1.2; future improvement is to thread the rationale through. Document this in the eval JSON output schema.

**Tests:**
- `build_eval_driver` returns a SmartMonsterDriver; sentinel deps raise on access.
- `run_scenario` with a mock AsyncOpenAI driver+judge returns a fully populated ScenarioResult.
- `run_scenario` handles driver returning a target that the judge rejects (None verdict) — `judge_error` populated.

**Verification:** `pytest tests/eval/test_runner.py -v` passes.

**Commit:** `feat(12-02): eval runner — driver/judge invocation bypassing Discord`
