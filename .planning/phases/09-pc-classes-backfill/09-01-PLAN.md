# Phase 9 Plan 01 — pc_classes Ingest-Backfill Script

## Objective

Ship a one-shot CLI tool (`eldritch-dm-backfill-pc-classes`) that walks all
existing `channel_sessions`, asks dm20 for each campaign's characters, and
populates the local `pc_classes` table. Closes TD-3 (silent no-Riposte-fires
gap) for v1.0 → v1.1 upgraders.

## Resolved Conflicts (resolved at PLAN time so executor doesn't stall)

| # | Conflict | Resolution |
|---|----------|------------|
| **C-1** | CONTEXT D-49 says "extract class + subclass" but `dm20__list_characters` returns no subclass field and `dm20__get_character` is unstructured text (see `persistence/pc_classes_repo.py` module docstring) | **Option (a) — best-effort.** Backfill `class_name` from `character_class` field; write `subclass=""`. Emit a per-row WARNING log so operators know which characters need hand-edits for subclass-gated features (Battle Master Riposte). Document in INSTALL.md upgrade flow. |
| **C-2** | REQUIREMENTS UPGRADE-01 calls for a PID-file lock check; CONTEXT omits it | **Defer.** v1.1 single-tenant assumption (D-48): operator knows whether the bot is running. Add a stderr WARNING when `--dry-run` is NOT set and we detect a live aiosqlite write lock (`sqlite3.OperationalError: database is locked`) — exits with code 3. PID-file logic deferred to v1.2. Documented in 09-01-SUMMARY. |
| **C-3** | `PCClassesRepo.upsert` SQL is `ON CONFLICT DO UPDATE`; REQUIREMENTS says `DO NOTHING` for idempotency | **Gate at the CLI, not the repo.** Default behavior: pre-check existence via `repo.get(channel_id, character_id)` and skip if non-None. `--force` flag skips the pre-check and lets the repo's `DO UPDATE` re-process. Repo SQL stays unchanged (other callers depend on update-on-conflict at ingest time). |
| **C-4** | CONTEXT D-43 says dry-run "opens SQLite read-only" but `PCClassesRepo` doesn't accept a URI/mode | Dry-run path **does not construct PCClassesRepo**. It opens `aiosqlite.connect("file:{path}?mode=ro", uri=True)` directly and does raw `SELECT character_id FROM pc_classes WHERE channel_id = ?`. Writes are driver-impossible. Real run uses `PCClassesRepo` as normal. |
| **C-5** | CONTEXT D-48 says dm20 fallback URL is `http://localhost:7777`; actual codebase default is `http://localhost:8765` (`settings.omlx_endpoint`) | Resolution order: `$DM20_MCP_URL` env → `$OMLX_ENDPOINT` minus `/v1` → `http://localhost:8765`. CONTEXT D-48 has the wrong default port; flagged as documentation drift in SUMMARY. |
| **C-6** | Test layout / asyncio mode | `pyproject.toml` has `asyncio_mode = "auto"` — tests are plain `async def`, no `@pytest.mark.asyncio` decorator. Match Phase 8's pattern. Mock dm20 via `respx` (already in `[dev]`). |

<task id="T-09-01-03" type="auto">
<title>SQLite upsert path + dry-run read-only + --force semantics</title>
<files>
  - src/eldritch_dm/tools/backfill_pc_classes.py  # add write path + dry-run
  - tests/tools/test_backfill_pc_classes.py  # add dry-run + force + idempotency tests
</files>
<actions>
  1. Add `async def apply_rows(rows, *, db_path, dry_run, force) -> ApplyReport`:
     - **dry_run=True path**: open `aiosqlite.connect(f"file:{db_path}?mode=ro", uri=True)`,
       per-row `SELECT 1 FROM pc_classes WHERE channel_id=? AND character_id=?`,
       classify as `would_insert | would_skip | would_update` based on existence
       and `force`. **Never constructs PCClassesRepo.** Driver-level write
       prohibition guaranteed by `mode=ro` (D-43).
     - **dry_run=False path**: construct `PCClassesRepo(db_path)`. For each row:
       if `force` → call `repo.upsert(...)` (DO UPDATE applies).
       else → `existing = await repo.get(channel_id, character_id)`; if None,
       call `repo.upsert(...)`; else skip + bump `skipped_existing` counter
       (C-3 gate; repo SQL unchanged).
  2. `ApplyReport` is a frozen pydantic v2 model with fields:
     `inserted: int, updated: int, skipped_existing: int, would_insert: int,
      would_skip: int, would_update: int, errors: list[str]`.
  3. Catch `sqlite3.OperationalError` containing "database is locked" → log
     ERROR with hint to stop the bot, return `EXIT_FATAL` (C-2 partial
     mitigation; full PID-file deferred to v1.2).
  4. Wire `main()` end-to-end:
     - parse args → `collect_rows()` → `apply_rows()` → compute exit code
       (D-44: 0 success, 2 partial if any failures from T-09-01-02, 3 fatal
       on DB lock).
     - Print final stdout summary table with structlog bound `dry_run`/`force`.
  5. Tests:
     - `test_dry_run_makes_no_writes` — uses a real temp sqlite + bootstrap;
       seed 0 pc_classes rows; collect 2 rows from respx; `dry_run=True`;
       assert post-state `SELECT COUNT(*) FROM pc_classes` == 0; assert
       report.would_insert == 2.
     - `test_dry_run_uses_readonly_uri` — patch `aiosqlite.connect` and
       assert the URI passed contains `mode=ro` and `uri=True`.
     - `test_real_run_inserts_new_rows` — seed empty pc_classes; run
       `dry_run=False`; assert 2 rows inserted; report.inserted == 2.
     - `test_idempotent_re_run_skips` — run twice; assert second run reports
       `skipped_existing == 2, inserted == 0`.
     - `test_force_re_processes_existing` — pre-populate 1 row with
       `class_name="stale"`; run with `force=True`; assert row's
       `class_name` matches the fresh dm20 value.
     - `test_db_locked_returns_exit_fatal` — patch repo.upsert to raise
       `sqlite3.OperationalError("database is locked")`; assert
       `main()` returns 3.
</actions>
