# Plan 17-02 — TTL fallback + `eldritch-dm-cache-clear` CLI + KPI emission

## Objective

Extend Plan 17-01's `CharacterCacheRepo` with:

1. A **TTL short-circuit** (D-123): `CHARCACHE_TTL_S` env (default 3600s). When an entry's `refreshed_ts + TTL > now`, `get_or_fetch` returns the cached snapshot **without** calling the fetcher — a true zero-network hit. Beyond TTL, fall through to the 17-01 ETag-refresh path.
2. A new operator CLI `eldritch-dm-cache-clear`, mirroring Phase 9's `eldritch-dm-backfill-pc-classes` shape (argparse, exit codes, structured logging). Wired into `pyproject.toml [project.scripts]`. Supports `--characters [--character-id ID]` and `--dry-run`.
3. KPI emission (D-126) via the Phase 11/13 dual-sink instrumentation pattern:
   - `eldritch.character_cache.lookup` per `get_or_fetch` call — attribute `cache.layer ∈ {"ttl_hit", "etag_match", "miss"}`, `cache.size`, `cache.latency_ms`.
   - `eldritch.character_cache.invalidation` per CLI clear / programmatic `invalidate` — attribute `invalidation.scope`, `invalidation.character_id`, `invalidation.entries_removed`.
   - Honors `OBSERVABILITY_ENABLED` (no module-level `opentelemetry` import).
4. A read-only `metrics_snapshot()` returning a frozen pydantic `CharacterCacheMetrics` with `hits_ttl`, `hits_etag`, `misses`, `size`, `invalidations_total`, `etag_match_rate` (computed).

## Known limitations (documented for the SUMMARY)

- **Level-up / equipment-change staleness window:** the cache caches semi-mutable static fields (level, equipment, base_stats). Without explicit per-mutation invalidation hooks at the dm20 call sites, TTL (default 3600s) is the sole freshness mechanism for level-ups. v1.5 ships with TTL-only; future phases MAY add explicit `cache.invalidate(character_id)` hooks at the level-up / shopping call sites. This is intentional v1.5 scope — operators can force-refresh via `eldritch-dm-cache-clear --characters --character-id ID`.
