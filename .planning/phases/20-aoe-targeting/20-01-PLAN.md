# Plan 20-01 — MonsterTacticChoice schema extension

## Objective

Extend `MonsterTacticChoice` (in `src/eldritch_dm/gameplay/smart_monster_driver.py`) from
single-target to multi-target while preserving **bit-identical behavior** for every
existing code path and test (v1.1 D-58 fail-soft, D-57 meta-knowledge guard).

After this plan: the schema accepts both legacy `{target_pc_id: "x"}` and new
`{target_pc_ids: ["x", "y"], tactic_kind: "aoe"}` JSON shapes; `choice.target_pc_id`
is a read-only `@property` returning the first element. Plan 02 builds on this
foundation with the prompt + ActionDescriptor + corpus expansion.

**Design (advisor-reviewed):** `target_pc_id` **cannot** be both a Field and a `@property`
on the same pydantic v2 model. The clean shape is:

```python
class MonsterTacticChoice(BaseModel):
    target_pc_ids: list[str]
    tactic_kind: Literal["single","aoe","multi_attack","breath","cone"] = "single"
    rationale: str | None = None

    @model_validator(mode="before")
    @classmethod
    def _coerce_legacy_shape(cls, data): ...  # {target_pc_id: "x"} → {target_pc_ids: ["x"]}

    @model_validator(mode="after")
    def _validate_kind_arity(self): ...       # single → len==1; aoe/breath/cone → len>=2

    @property
    def target_pc_id(self) -> str:
        return self.target_pc_ids[0]
```

This keeps every existing call site working unchanged (D-149/D-155 backwards-compat).
