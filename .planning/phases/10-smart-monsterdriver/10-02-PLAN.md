### Task 1 ‚Äî Per-round cache

<task type="auto" tdd="false">
<intent>
Cache LLM decisions keyed on `(channel_id, round_number, monster_id)`. Hits
return the cached `MonsterTacticChoice` without re-calling the LLM. Cache
lives on the driver instance; eviction is implicit when the round advances
(stale keys remain in dict but are never read because the key tuple changes).
A `_cache_max_size = 256` guard evicts the oldest entry FIFO to avoid
unbounded growth in long sessions.
</intent>

<files>
- src/eldritch_dm/gameplay/smart_monster_driver.py (extend)
</files>

<behavior>
`self._cache: collections.OrderedDict[tuple[str, int, str], MonsterTacticChoice] = OrderedDict()`.
In `_pick_target_llm`:
  - Compute `key = (channel_id, round_number, monster_id)`
  - If `key in self._cache`: log `event=smart_driver_cache_hit`, return cached choice
  - On successful LLM resolution: `self._cache[key] = choice`, evict oldest if `len > 256`
</behavior>

<verify>
Test: call `_pick_target_llm` twice with the same key ‚Äî mock client called
exactly once. Test: 257 distinct keys ‚Üí oldest evicted, latest cached.
</verify>

### Task 2 ‚Äî Factory + MONSTER_DRIVER env var

<task type="auto" tdd="false">
<intent>
Centralize driver construction. The orchestrator should never instantiate
`MonsterDriver` or `SmartMonsterDriver` directly ‚Äî only via
`make_monster_driver(...)`.
</intent>

<files>
- src/eldritch_dm/gameplay/monster_driver_factory.py (NEW)
- src/eldritch_dm/config/__init__.py (add `monster_driver: Literal["smart","random","mixed"] = "smart"` field, alias `MONSTER_DRIVER`)
</files>

<behavior>
`make_monster_driver(*, env_override: str | None = None, **driver_kwargs) -> MonsterDriver`:
  - resolution order: `env_override` arg > `Settings().monster_driver` > `"smart"` default
  - `"random"` ‚Üí returns `MonsterDriver(**driver_kwargs)` (v1.0 behavior, escape hatch)
  - `"smart"` ‚Üí returns `SmartMonsterDriver(**driver_kwargs)`
  - `"mixed"` ‚Üí returns `SmartMonsterDriver(**driver_kwargs)` (the routing inside
    `_route_path` already implements the mixed semantics; "mixed" at the factory
    level just confirms the smart driver is used ‚Äî the per-monster INT-gating
    handles the rest)
  - unknown value ‚Üí log warning, fall back to `"smart"`

The `Settings` field uses the same alias pattern as `eligibility_yaml_path`
(`alias="MONSTER_DRIVER"`) so it picks up the bare env var name.
</behavior>

<verify>
- `make_monster_driver(env_override="random", **kwargs)` returns `MonsterDriver`
- `make_monster_driver(env_override="smart", **kwargs)` returns `SmartMonsterDriver`
- `make_monster_driver(env_override="garbage", **kwargs)` returns `SmartMonsterDriver` + logs warning
- With `MONSTER_DRIVER=random` in monkeypatched env, no override ‚Üí returns `MonsterDriver`
</verify>

### Task 3 ‚Äî Adversarial corpus (‚â•15 scenarios)

<task type="auto" tdd="false">
<intent>
New file `tests/gameplay/test_monster_driver_corpus.py` exercising the full
fail-soft envelope. Mirrors the v1.0 sanitizer corpus pattern (one test class,
one parametrized scenario list, each scenario asserts no exception leaks AND
the driver still calls `next_turn` to avoid deadlock).
</intent>

<files>
- tests/gameplay/test_monster_driver_corpus.py (NEW)
</files>

<behavior>
Each scenario builds a SmartMonsterDriver with a stub `AsyncOpenAI` client
whose `chat.completions.create` is an AsyncMock with a custom `side_effect`.
The 15 scenarios (each is one test function ‚Äî easier to read than one big
parametrize):

1. `test_corpus_malformed_json` ‚Äî mock returns string `"not json at all"` ‚Üí random fallback
2. `test_corpus_hallucinated_target_id` ‚Äî JSON valid, id not in candidates ‚Üí random fallback
3. `test_corpus_timeout_exceeded` ‚Äî `side_effect=asyncio.TimeoutError()` ‚Üí random fallback + warning logged
4. `test_corpus_empty_candidate_list` ‚Äî pcs=[] in state ‚Üí no LLM call, `next_turn` called, no crash
5. `test_corpus_sub_int_bypass` ‚Äî monster INT=2 ‚Üí LLM never called (assert mock.call_count == 0)
6. `test_corpus_int12_with_downed_pc` ‚Äî INT=12, one PC at hp_current=0 ‚Üí no exception (anti-griefing is an aspirational signal, not a hard guarantee; just assert no crash)
7. `test_corpus_int18_with_concentration_holder` ‚Äî INT=18, one PC has `active_conditions=["concentrating"]` ‚Üí no exception, LLM is called
8. `test_corpus_invisible_pc` ‚Äî PC has `active_conditions=["invisible"]` ‚Üí no exception
9. `test_corpus_refusal_path` ‚Äî mock returns content `""` (empty) ‚Üí random fallback
10. `test_corpus_rate_limit_429` ‚Äî `side_effect=openai.RateLimitError(...)` ‚Üí random fallback
11. `test_corpus_cache_hit_same_key` ‚Äî call twice with same (c,r,m); mock called once
12. `test_corpus_pc_death_between_calls` ‚Äî call once, mutate candidate list to remove the previously-chosen PC, call again with different round (so cache miss); ensure new candidate set respected
13. `test_corpus_cross_channel_isolation` ‚Äî same (round, monster_id) different channel_id ‚Üí mock called twice (cache miss)
14. `test_corpus_mixed_mode_seeded_determinism` ‚Äî INT=6, same (c,r,m) ‚Üí same route choice across two evaluations
15. `test_corpus_self_target_attempt` ‚Äî LLM returns the monster's own id ‚Üí random fallback (id not in PC candidate set)

ALL 15 must:
  - assert no exception propagated
  - assert `next_turn` was eventually invoked (combat keeps moving)
</behavior>

<verify>
`pytest tests/gameplay/test_monster_driver_corpus.py -v` ‚Äî 15 passing.
</verify>
