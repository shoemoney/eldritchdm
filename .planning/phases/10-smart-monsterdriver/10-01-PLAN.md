# Phase 10 Plan 01 — SmartMonsterDriver core (INT-gating, LLM oracle, 1500ms deadline)

## Objective

Ship `src/eldritch_dm/gameplay/smart_monster_driver.py` exporting `SmartMonsterDriver`,
a sibling to v1.0's `MonsterDriver`. The smart driver replaces only the
`_pick_target` step: it consults the local oMLX/ShoeGPT LLM (via the existing
`AsyncOpenAI` client pattern) for INT≥8 monsters, falls back to random for
INT≤4, and 50/50-samples for INT in [5,7] using a deterministic seed. The
whole smart path is bounded by a 1500ms `asyncio.wait_for`; ANY exception →
random fallback (fail-soft). Pydantic `MonsterTacticChoice` validates JSON;
hallucinated `target_pc_id`s trigger fallback (not exceptions).

## Context

- @.planning/phases/10-smart-monsterdriver/10-CONTEXT.md (locked decisions D-50..D-61)
- @.planning/phases/10-smart-monsterdriver/10-AI-SPEC.md (LLM design contract)

- @src/eldritch_dm/ingest/translate.py (existing `response_format={"type":"json_object"}` + post-parse validation pattern)
- @src/eldritch_dm/config/__init__.py (env var + Settings pattern)

## Critical reconciliations (CONTEXT D-50/D-51)

The AI-SPEC draft assumes hosted OpenAI with
`.beta.chat.completions.parse(response_format=PydanticModel)`. **Production
uses local oMLX/ShoeGPT, which is unreliable in strict-parse mode.** This
plan uses:

```python
response_format={"type": "json_object"}
# then:
choice = MonsterTacticChoice.model_validate_json(content)
```

If `model_validate_json` raises, attempt one regex-based fallback extractor
(`re.search(r'"target_pc_id"\s*:\s*"([^"]+)"', content)`) before giving up
to random.

### Task 5 — Wire `_pick_target` to call route + integrate fallback

<task type="auto" tdd="false">
<intent>
Public override: SmartMonsterDriver.drive() either re-implements the v1.0 flow
with the new pick step OR (preferred) the v1.0 driver gains an injectable
`pick_strategy` callable and SmartMonsterDriver injects its smart picker. To
keep the diff scoped, this plan takes the **subclass-with-override** route:
SmartMonsterDriver subclasses MonsterDriver, overrides `_choose_target` (a
new extracted method on MonsterDriver) — see Task 5a for the v1.0 refactor.
</intent>

<files>
- src/eldritch_dm/gameplay/monster_driver.py (extract `_choose_target` method)
- src/eldritch_dm/gameplay/smart_monster_driver.py (override `_choose_target`)
</files>

<behavior>
**5a (refactor — non-behavioral):** In `monster_driver.py`, extract the
existing inline `self._random_choice(targets)` line into
`self._choose_target(targets, *, channel_id, round_number, current_actor)`.
Default implementation returns `self._random_choice(targets)`. Behavior is
identical — v1.0 tests must still pass.

**5b:** In `smart_monster_driver.py`, override `_choose_target` to:
  1. Extract `monster_int` from `current_actor.get("intelligence")` (or
     `current_actor.get("stats", {}).get("intelligence")` — try both)
  2. Compute route via `_route_path`
  3. If route ∈ {"random", "mixed_random"} → return `self._random_choice(targets)`
  4. If route ∈ {"llm", "mixed_llm"}:
     a. await `self._pick_target_llm(...)` — returns PC dict or None
     b. If None → log path=random_fallback, return `self._random_choice(targets)`
     c. Else → log path=smart_ok, return the LLM-picked dict

NOTE: `_choose_target` must be async to support the LLM await. Update the
sync `_random_choice` call site in `monster_driver.py` to `await
self._choose_target(...)`.
</behavior>
