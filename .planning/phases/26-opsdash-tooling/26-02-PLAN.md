## Tasks

### Task 1 (auto) — Rewrite `scripts/audit/backfill_summary_frontmatter.py`

**Files:**
- `scripts/audit/backfill_summary_frontmatter.py` (rewrite — drop the
  hardcoded `MAPPING` constant).

**Implementation:**
- Discovery: `pathlib.Path(".planning/phases").rglob("*-SUMMARY.md")`.
- For each discovered SUMMARY (e.g. `16-01-SUMMARY.md`):
  - Compute the sibling PLAN filename by replacing `-SUMMARY.md` →
    `-PLAN.md` in the same directory.
  - If no sibling PLAN exists, emit `WARN ... no sibling PLAN` and skip
    (do not modify).
  - Parse the PLAN's YAML frontmatter (`--- ... ---` block at top).
    Extract `requirements:` — accept flow form (`[FOO-01, FOO-02]`) only
    (sufficient for current corpus; emit `WARN ... unparseable
    requirements` and skip on anything else).
  - Treat empty list / missing field as `[]` and still emit
    `requirements_completed: []` (Phase 14 contract — the key must be
    present for the CI gate; an empty list explicitly records "this plan
    completed nothing requirement-tracked").
- Frontmatter rewrite logic from the existing script is preserved
  verbatim (`_split_frontmatter`, `_backfill_frontmatter`,
  `_format_value`, `_process_file`). Only the driver (`main`) and the
  derivation of the per-SUMMARY req list change.
- Flags: `--dry-run` (default) prints unified diffs;
  `--apply` rewrites in place.
- Exit non-zero only on hard error (unreadable file, malformed
  frontmatter in a PLAN we tried to read).

**Behavior contract:**
- Running `--dry-run` against the current repo SHOULD propose edits for
  every SUMMARY whose `requirements_completed:` value is currently empty
  (14 candidates per discovery as of 2026-05-25 — phases 16-22 and one
  legacy plan that ticked WIRE-02 only when its sibling PLAN's
  `requirements:` field had `[WIRE-02]` — i.e. honour the PLAN, not the
  ROADMAP table).
- Running `--apply` then `--dry-run` again MUST report 0 changes — this
  is the OPSDASH-02 "0 gaps to apply" validation.

**Done when:** the new script imports cleanly, `--dry-run` emits diffs for
the empty SUMMARYs, `--apply` rewrites them, and a follow-up `--dry-run`
is a no-op.

### Task 3 (auto) — Extend `.planning/UPSTREAM-ISSUES.md` with ISSUE-2 and ISSUE-3

**Files:**
- `.planning/UPSTREAM-ISSUES.md` (append two entries).

**Implementation:**

**ISSUE-2: `backfill_summary_frontmatter.py` hardcoded path table (RESOLVED in-repo)**

- Target project: gsd-tools (executor/audit-script template).
- Severity: medium (traceability silently drifts when audit script
  isn't kept in sync with new phases).
- Repro: any executor that writes a SUMMARY without populating
  `requirements_completed:`, paired with an audit script that
  hardcodes the {SUMMARY → req-list} mapping, will drop coverage when
  new phases ship.
- Evidence: Phase 24 audit found 14 SUMMARYs (phases 16-22) whose
  `requirements_completed:` was empty because the hardcoded `MAPPING`
  in `backfill_summary_frontmatter.py` only covered phases 6-13.
- Resolution: OPSDASH-02 rewrote the script to auto-discover SUMMARYs
  and infer requirements from sibling PLAN frontmatter. Entry stands
  as historical proof and template for future audit-script regressions.
- Status: **RESOLVED IN-REPO** (no upstream filing needed unless gsd-tools
  ships a similar audit-script generator).

**ISSUE-3: dm20 MCP lacks structured post-resolve damage events**

- Target project: dm20 MCP server.
- Severity: high (blocks EldritchDM's WIRE-01 narration-quality work).
- Repro: invoke any dm20 combat-resolution tool that applies HP
  changes. The tool returns the post-state but emits no structured
  event describing the delta (damage amount, source, target, damage
  type). Consumers must diff pre/post snapshots to reconstruct the
  event — error-prone and loses type info.
- Evidence: EldritchDM v1.7 WIRE-01 (Phase 23) needed
  `post_resolve_damage_events` to feed the narration prompt; the
  workaround was a Python-side diff that loses damage-type fidelity
  for resistances/immunities. The deferred half of Phase 23 (the
  "concentration" check) hit the same gap.
- Suggested fix: dm20 emits a structured event list alongside the
  resolved state: `[{"target": ..., "amount": ..., "type": ...,
  "source_actor": ..., "source_action": ...}, ...]`.
- Status: **OPEN** — requires upstream dm20 work; tracked here so we
  re-evaluate at the next dm20 version bump.

**Done when:** the file has three top-level `## ISSUE-N` sections; the
existing ISSUE-1 is untouched.
