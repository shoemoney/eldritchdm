# Phase 13 Plan 02 — alerts.yaml + Degraded-Mode Trigger + Auto-Recover

## Objective

Land the alerting control-plane: a `database/alerts.yaml` driven by the same
3-tier loader pattern as Phase 8, a small **AlertEvaluator** that polls
`compute_kpis()` (Plan 01) and fires actions on rule match, and the
**degraded-mode** state machine that flips the monster driver to `random` when
P99 latency breaches 1500ms for 5 consecutive minutes and auto-recovers when
P99 returns below 1200ms for 5 consecutive minutes (hysteresis). After this
plan, MON-02 is complete.

## Architectural Decisions (refine CONTEXT given code reality)

| Ref | Refinement | Why |
|---|---|---|
| **R-13-02-a** | **Degraded mode is module-level state in `observability/alerts.py`, NOT a `Settings` field.** `Settings` is `frozen=True` (immutable). We expose a tiny `DegradedModeState` singleton with `active: bool`, `reason: str | None`, `entered_at_utc: datetime | None`, and thread-safe `trip(reason)` / `recover()` methods. `monster_driver_factory._resolve_mode()` gains a single new line: `if degraded_mode.is_active(): return "random"` placed BEFORE the env lookup. | `Settings.frozen=True` is non-negotiable (immutability invariant from Phase 1). Module-level state is the standard pattern when you need cross-asyncio-task mutability. |
| **R-13-02-b** | **AlertEvaluator runs as a background asyncio task**, scheduled by the bot's existing startup. Tick interval: 30s (configurable via `ELDRITCH_ALERT_TICK_SECONDS`). On each tick: call `compute_kpis()`, evaluate each rule, accumulate per-rule "consecutive breach minutes" counters in memory. Rule fires when its `window_minutes` threshold is hit. State machine survives bot restarts ONLY via the span buffer (replay last 5min of spans on cold start to determine if we should re-enter degraded mode immediately). | Same async lifecycle as discord.py's task loops; no new scheduler dep. |
| **R-13-02-c** | **alerts.yaml schema** mirrors `eligibility.yaml`: `version: 1`, `rules: [...]`. Each rule = `{name, severity, kpi, op, threshold, window_minutes, action, routing}`. Validated by pydantic v2 `AlertsFile` with `extra='forbid'`. `safe_load` only (CI grep gate from Phase 8 already enforces — we ride that gate). Fail-soft to a built-in `DEFAULT_RULES` frozenset on any error (same pattern as Phase 8). | Reuses the proven loader pattern, including the YAML-1 RCE guard. |
| **R-13-02-d** | **Hysteresis test uses an injectable `time_source` callable**, NOT `freezegun`. Default `time_source = lambda: datetime.now(UTC)`. Test passes a mutable clock through the evaluator constructor and advances it manually. Avoids freezegun's known races with `threading.Event` (which we'd hit in `span_buffer`'s background thread). | Same test pattern Phase 10 used for `asyncio.wait_for` timeouts. Stable on CI. |
| **R-13-02-e** | **Routing actions for v1.2**: `log` (structlog event with rule context) + `degrade` (calls `degraded_mode.trip(reason)`). `webhook` and `throttle` are accepted by the schema but emit a `WARNING` structured-log saying "v1.3 routing not yet implemented" and otherwise no-op. CONTEXT deferred webhook routing to v1.3 explicitly. | Schema is forward-compatible; impl is honest about what works today. |
| **R-13-02-f** | **Cold-start replay**: on bot boot, AlertEvaluator first calls `compute_kpis(window_minutes=5)` from the buffer (which has up to 5min of pre-restart data thanks to D-92 persistence). If the result already trips a `degrade` rule, set degraded mode immediately so a bot restart during a real outage doesn't briefly hand a smart-driver call out. | Operationally correct; matches AI-SPEC's "5 consecutive minutes" intent across process boundaries. |

<task id="04" type="auto">
  <title>AlertEvaluator (rule engine + injectable time_source + hysteresis recover)</title>
  <files>
    <create>src/eldritch_dm/observability/alert_evaluator.py</create>
    <create>tests/observability/test_alert_evaluator.py</create>
  </files>
  <behavior>
    `AlertEvaluator` constructor:
    ```python
    AlertEvaluator(
        rules: Sequence[AlertRule],
        *,
        kpi_provider: Callable[[], KPISnapshot] = compute_kpis,
        time_source: Callable[[], datetime] = lambda: datetime.now(UTC),
        recover_threshold_factor: float = 1200/1500,  # AI-SPEC §7 hysteresis
    )
    ```
    Public methods:
      - `tick() -> AlertEvaluationResult` — pulls one KPI snapshot, evaluates every rule:
          * For each rule R with action=='degrade', track per-rule consecutive-breach count
            (incremented when condition true at this tick, reset to 0 when false).
            When consecutive count * tick_interval ≥ window_minutes*60 → fire `degrade`.
          * For action='log': emit `eldritch.alert.fired` at the rule's severity log
            level on every tick where condition is true (no consecutive-count gating
            for log-only rules — operators want to see them as they happen).
          * For action='throttle' / 'webhook': emit `eldritch.alert.deferred` WARNING
            "v1.3 routing not yet implemented".
      - Hysteresis recover: for each rule that previously tripped `degrade`, check the
        same KPI against `threshold * recover_threshold_factor` with the opposite
        operator. When that "recover threshold" is satisfied for `window_minutes`
        consecutive ticks, call `degraded_mode.recover()`. The 1200/1500 factor is
        the AI-SPEC §7 default; rules can override via an optional `recover_threshold`
        field on AlertRule (deferred to a later refinement — keep factor-based for v1.2).
      - `start(loop, tick_seconds: float = 30.0) -> asyncio.Task` — schedules
        `tick()` on the running event loop with the configured interval. Tick errors
        are caught and logged but NEVER crash the loop.
    Tests use a synthetic `time_source` and a stubbed `kpi_provider` returning a
    sequence of snapshots that crosses thresholds. Assert: trip occurs at 5min
    breach, NOT at 4:59min; recover occurs at 5min cooldown ≤1200ms, NOT at 4:59min;
    log-only rules do not call degrade(); throttle/webhook emit deferred warning.
  </behavior>
  <verification>`pytest tests/observability/test_alert_evaluator.py -q` passes.</verification>
  <done>Trip + recover hysteresis verified deterministically; log/degrade actions correct.</done>
