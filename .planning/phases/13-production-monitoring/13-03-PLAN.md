# Phase 13 Plan 03 — Cost Guard + Daily Budget Cap + eldritch-dm-cost-report CLI

## Objective

Land the cost-control plane: a **cost calculator** mapping
`(model_name, tokens_input, tokens_output) → USD` driven by
`database/pricing.yaml` (3-tier loader), a **daily-budget guard** that flips
degraded mode when `ELDRITCH_DAILY_LLM_BUDGET_USD` is breached, and the
`eldritch-dm-cost-report` CLI for offline daily-spend reports. After this
plan, MON-03 is complete and Phase 13 closes.

## Architectural Decisions (refine CONTEXT given code reality)

| Ref | Refinement | Why |
|---|---|---|
| **R-13-03-a** | **Pricing values are fetched JUST-IN-TIME during execution via Tavily/Brave research**, recorded in `database/pricing.yaml` with `source_url:` + `as_of:` comments per entry. Plans cannot bake 2026 numbers without active research — pricing changes monthly. The unit-test budget asserts ±5% against the values committed in the YAML (NOT against a hardcoded constant), so when operators update pricing.yaml the tests don't suddenly fail. | Per CLAUDE.md `<key_pitfalls_to_avoid>` #6 — pricing is operator-visible and source-tracked. |
| **R-13-03-b** | **Cost calculator is pure Python**: `calculate_cost(model: str, tokens_input: int, tokens_output: int, *, table: PricingTable) -> Decimal`. Returns `Decimal` (NOT float) to keep cumulative sums exact for the daily total. Unknown model → log `eldritch.cost.unknown_model` WARNING + return `Decimal(0)` (cost report explicitly flags any unknown-model spans so operators can audit). | Float arithmetic on currency drifts over a day's worth of spans. |
| **R-13-03-c** | **Daily budget guard is a new alert action**, NOT a separate evaluator. We add `action="cap_budget"` to the alerts.yaml schema (extending Plan 02's enum) and ship a default rule in `pricing.yaml`'s neighbor file — actually no, simpler: the cost guard is its own `BudgetEvaluator` class with the SAME injectable `time_source` pattern as `AlertEvaluator`. It ticks every 5 minutes (configurable), sums today's spans into USD, and calls `degraded_mode.trip("budget_exceeded:$X over $Y")` on breach. It also logs at $2/day per AI-SPEC §6 ("alert at $2/day"). Keeps alert engine focused on KPI thresholds; cost is its own concern with different cadence and aggregation. | Adding action="cap_budget" to alerts.yaml conflates rate-of-error with cumulative-spend semantics. Separate evaluator is clearer. |
| **R-13-03-d** | **Today's window**: "today" = UTC date of the configured `time_source()`. Operators in non-UTC timezones get a midnight-UTC rollover, which is the documented convention. CLI `--since` flag accepts any ISO-8601 timestamp for backfills. | Avoids tz-aware datetime tangles; UTC is the bot's wire-format anyway. |
| **R-13-03-e** | **Cost-report CLI exit codes** mirror Phase 9's pattern: 0=ok, 1=user error (bad flags, missing buffer), 2=partial (some spans had unknown models — report emitted with warning section), 3=fatal (buffer corrupt). | Operator-script-friendly; consistent with eldritch-dm-backfill-pc-classes. |
| **R-13-03-f** | **5-workload test corpus** for ±5% accuracy: (1) small-chat 100in/200out gpt-4o-mini, (2) large-chat 4000in/800out gpt-4o, (3) judge-eval 1500in/400out claude-sonnet-4-6, (4) cheap-judge 1500in/400out claude-haiku-4-5-20251001, (5) local ShoeGPT 1000in/500out (=$0). Each workload has a hand-computed `expected_usd` AS A COMMENT in the test file; the test asserts within ±5%. When operators update pricing.yaml they MUST update the hand-computed expected values in the test — intentional friction. | CONTEXT explicitly named ±5% across 5 workloads; the test makes pricing-table updates auditable. |
