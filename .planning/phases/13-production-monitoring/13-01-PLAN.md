# Phase 13 Plan 01 — KPI Monitors + Span Buffer + Prometheus /metrics

## Objective

Land the data-plane for production monitoring: a local SQLite **span buffer** that
becomes the single source of truth for every traced decision (independent of
Phoenix/OTLP), a 5-minute rolling **KPI computer** that derives the 5 AI-SPEC §7
indicators from the buffer, and an opt-in **Prometheus `/metrics` endpoint** at
`:9090/metrics` exposing those KPIs as gauges/counters. After this plan, MON-01
is complete and 13-02 / 13-03 can plug into the buffer + KPI APIs.

## Architectural Decisions (refine CONTEXT given code reality)

| Ref | Refinement | Why |
|---|---|---|
| **R-13-01-a** | **Span buffer is the PRIMARY sink**, OTLP is a SECONDARY sink. The `traced_*` context managers in `instrumentation.py` are modified so the **`__exit__`** path always writes a row to `span_buffer.sqlite` regardless of `_TRACER` state. OTel export remains gated by `OBSERVABILITY_ENABLED=true`. The KPI monitors read exclusively from the buffer; they never call OTel APIs. | D-83 says "monitors must run even without Phoenix". Piggybacking on the existing context managers avoids touching all 3 call sites in `smart_monster_driver`, `ingest/translate`, `eval/judge`. |
| **R-13-01-b** | **Buffer writes go through a small in-process queue + background flush thread**, NOT an `await aiosqlite` call inside the context manager. The hot path appends to a `queue.Queue`; a daemon thread drains it into SQLite with `WAL` mode, batch-committing every 100 rows or 500ms. | Keeps the bot event loop unblocked even when SQLite is slow (lock contention, disk pressure). Same pattern as `structlog`'s queue handler. |
| **R-13-01-c** | **Use `prometheus_client.start_http_server(port=9090)`**, NOT `aiohttp.web`. CONTEXT D-84 mis-stated the server lib; CLAUDE.md explicitly forbids `aiohttp`. `prometheus_client` ships its own stdlib `http.server`-based scrape endpoint that runs in a background thread. | Honors stack constraints; zero new heavy deps. |
| **R-13-01-d** | Add `prometheus_client>=0.20,<1.0` to the existing `[project.optional-dependencies] observability` group from Phase 11. Module imports remain lazy — `metrics_endpoint.py` only imports `prometheus_client` inside `start_metrics_endpoint()` after checking the env gate. The lazy-import canary in `test_lazy_import.py` continues to pass because it asserts only on `opentelemetry`. We add a sibling canary that also asserts no `prometheus_client` in `sys.modules` when the endpoint env is off. | Consistent with Phase 11's "off by default, zero cost" pattern. |
| **R-13-01-e** | **KPI snapshot model**: `KPISnapshot` (pydantic v2 dataclass) with the 5 fields from D-85. Computed via a single `compute_kpis(now: datetime, window_minutes: int = 5) -> KPISnapshot` SQL aggregation against the buffer. Result is also cached in-process for 5s to avoid hammering SQLite on every Prometheus scrape. | A Prometheus scrape every 15s should not trigger 5 distinct quantile recomputations. |

<task id="02" type="auto">
  <title>Span buffer SQLite storage layer (~/.eldritch/spans.sqlite, WAL, 7-day retention)</title>
  <files>
    <create>src/eldritch_dm/observability/span_buffer.py</create>
    <create>tests/observability/test_span_buffer.py</create>
  </files>
  <behavior>
    Module exposes:
      - `BufferRow` (pydantic) — schema: `(timestamp_utc TEXT, span_name TEXT, monster_id TEXT|NULL, channel_id TEXT|NULL, combat_round INTEGER|NULL, driver_path TEXT|NULL, latency_ms INTEGER|NULL, tokens_input INTEGER|NULL, tokens_output INTEGER|NULL, fallback_reason TEXT|NULL, model TEXT|NULL, scenario_id TEXT|NULL, overall_score REAL|NULL, refusal BOOLEAN, error TEXT|NULL)`.
      - `init_buffer(path: Path | None = None) -> SpanBuffer` — lazy singleton. Module-level `_BUFFER` sentinel under a one-time `threading.Lock`; **on hot path (subsequent calls) returns immediately — NO mkdir, NO sqlite3.connect, NO schema check.** Only the first call creates the parent dir, opens stdlib `sqlite3` with `journal_mode=WAL`, `synchronous=NORMAL`, `busy_timeout=5000`, creates `spans` table + index `(timestamp_utc, monster_id, channel_id)`. Path default: `~/.eldritch/spans.sqlite`; overridable via `ELDRITCH_SPAN_BUFFER_PATH` env (test fixtures use tmp_path). A `reset_for_tests()` helper clears the sentinel.
      - `SpanBuffer.record(row: BufferRow) -> None` — appends to in-process `queue.Queue`, non-blocking, drops + structlog-warns if queue >10k entries (pathological-load safety valve). After enqueue, fires registered observer callbacks synchronously (each in a try/except so a misbehaving observer cannot break the hot path).
      - `SpanBuffer.add_post_write_observer(observer: Callable[[BufferRow], None]) -> None` — inversion-of-control hook so external modules (e.g. `metrics_endpoint` registering a Prometheus Counter increment) can react to writes WITHOUT `span_buffer` importing their libs. Crucial for the lazy-import canary: `span_buffer` itself never imports prometheus_client; the observer is registered only when `start_metrics_endpoint()` runs (i.e. env gate is on).
      - Background daemon thread pulls from queue, batches up to 100 rows or 500ms, commits.
      - `SpanBuffer.flush(timeout_s: float = 2.0) -> None` — synchronously drains queue (used by tests + KPI computer before reading).
      - `SpanBuffer.prune_older_than(days: int = 7) -> int` — bulk DELETE; returns row count removed. Called by `init_buffer` on first open (lazy startup prune) and by a scheduled task in 13-02.
      - `SpanBuffer.query(since: datetime, until: datetime | None = None, span_name: str | None = None) -> list[BufferRow]` — read API used by KPI computer + cost report.
    Tests cover: schema create idempotent, WAL mode active, record→flush→query round-trip, prune removes rows older than N days, queue overflow drops + logs (does not raise), parent dir auto-created, threadsafe under 1000 concurrent records, observer callbacks fire post-write (and a raising observer is logged but does not propagate), `init_buffer()` hot-path early-return verified by patching `sqlite3.connect` to raise on second call.
  </behavior>
  <verification>`pytest tests/observability/test_span_buffer.py -q` passes.</verification>

  <title>KPI computer (5 KPIs over rolling 5-min window, in-process 5s cache)</title>
  <files>
    <create>src/eldritch_dm/observability/kpi.py</create>
    <create>tests/observability/test_kpi.py</create>

  <behavior>
    Exposes:
      - `KPISnapshot` (pydantic frozen) with fields:
          `latency_p99_ms: float | None`,
          `success_rate: float | None`,
          `tactical_score: float | None`,   # nan-equivalent represented as None
          `refusal_rate: float | None`,
          `fallback_rate: float | None`,
          `window_seconds: int`,
          `sample_size: int`,
          `computed_at_utc: datetime`.
      - `compute_kpis(now: datetime | None = None, window_minutes: int = 5) -> KPISnapshot`
        Queries the buffer for the 5-min window ending at `now`, runs SQL aggregations:
          * p99 latency via sorted-percentile over `latency_ms` from `eldritch.monster.decision` spans.
          * success_rate = count(driver_path='smart' AND fallback_reason IS NULL) / count(driver_path IN ('smart','random','mixed')).
          * tactical_score = avg(overall_score) over `eldritch.eval.judge` spans within window; None if no eval rows.
          * refusal_rate = count(fallback_reason='refusal') / count(*).
          * fallback_rate = count(fallback_reason IS NOT NULL) / count(*).
        If `sample_size == 0`, all 5 KPI fields are `None`.
      - `get_cached_kpis(ttl_seconds: float = 5.0) -> KPISnapshot` — module-level cache (threading.Lock-guarded) avoiding recomputation on each Prometheus scrape (R-13-01-e).
    Tests use a tmp-path buffer fixture, seed synthetic rows, assert each KPI computation against hand-computed expected values. Edge cases: empty window, all-fallback window, mixed driver_paths, nan tactical_score (no eval rows present).
  </behavior>
