# Phase 22 Plan 02 — Discord DM-to-owner + Phase 16↔17 invalidation wire

## Objective

1. (OPQOL-02) Operator gets a Discord DM when budget breaches and/or
   degraded mode transitions. Opt-in via `DISCORD_OWNER_ID`; rate-limited
   to 1 DM per event-type per hour.
2. (OPQOL-03) When dm20 schema version changes, BOTH MCP cache AND
   character cache get wiped. Partial wipe (one succeeds, one fails) is
   acceptable — log `eldritch.cache.partial_wipe` and continue.

### Task 1 — Add `notify_callbacks` hook to `DegradedModeState`
<task type="auto">
<files>src/eldritch_dm/observability/degraded_mode.py</files>

<behavior>
Extend `DegradedModeState`:

```python
NotifyEvent = Literal["degraded_mode_entered", "degraded_mode_exited"]
NotifyCallback = Callable[[NotifyEvent, str | None], None]
# event, reason (None on exit)

def add_notify_callback(self, cb: NotifyCallback) -> None: ...
def remove_notify_callback(self, cb: NotifyCallback) -> None: ...
```

Behavior:
- Callback list held under the existing `_lock`.
- Inside `trip()`, after the first-time-entered log (NOT on idempotent
  re-trips that only change reason), iterate callbacks, invoke each in a
  try/except that logs `eldritch.degraded_mode.notify_error` and continues.
  This means one bad callback never blocks others or raises out of `trip()`.
- Inside `recover()`, after the `exited` log, same iteration.
- Reason-change re-trips do NOT fire callbacks (avoid spamming the owner
  every time the reason string changes).
- Callbacks are invoked SYNCHRONOUSLY inside trip/recover. Discord IO has
  to happen on the event loop, so the callback's job is to schedule via
  `asyncio.run_coroutine_threadsafe` or `loop.call_soon_threadsafe`. The
  BudgetOwnerNotifier (Task 2) handles that scheduling internally.

Add `reset_for_tests()` to also clear callbacks.
</behavior>

### Task 2 — New module `observability/budget_dm.py` (`BudgetOwnerNotifier`)
<task type="auto">
<files>src/eldritch_dm/observability/budget_dm.py</files>

<behavior>
New module. Public surface:

```python
EventType = Literal["budget_breached", "degraded_mode_entered", "degraded_mode_exited"]

class BudgetOwnerNotifier:
    def __init__(
        self,
        *,
        bot: Any,                              # discord.Client-like; needs fetch_user
        owner_id: int | None,
        loop: asyncio.AbstractEventLoop | None = None,
        rate_limit_window_s: float = 3600.0,
        clock: Callable[[], float] | None = None,
    ) -> None: ...

    # Sync entrypoint — safe from BudgetEvaluator.tick (thread executor)
    # and from DegradedModeState.trip/recover (sync, possibly cross-thread).
    def notify(self, event: EventType, reason: str | None = None) -> None: ...

    # Async — used in tests + direct call sites that already have the loop.
    async def notify_async(self, event: EventType, reason: str | None = None) -> None: ...

    # Convenience: register self as a degraded_mode callback.
    def attach_to_degraded_mode(self) -> None: ...
    def detach_from_degraded_mode(self) -> None: ...
```

Semantics:
- `owner_id is None` → every `notify` is an immediate no-op.
- In-memory `dict[EventType, float]` keyed by event-type tracks last-sent
  timestamp. If `clock() - last_sent < rate_limit_window_s` → swallow with
  `eldritch.budget_dm.rate_limited` log, no DM.
- `notify(sync)`: if no loop provided, get the running loop; if no running
  loop is available, fall back to `asyncio.get_event_loop_policy().get_event_loop()`
  → on any loop-acquisition failure, log + no-op (NEVER raise).
- Schedule `notify_async` onto the loop via `loop.call_soon_threadsafe(
  lambda: loop.create_task(self.notify_async(event, reason)))`.
- `notify_async`: build a short message (1–3 lines), `user = await
  self._bot.fetch_user(self._owner_id)`, `await user.send(msg)`. Swallow:
  `discord.Forbidden`, `discord.NotFound`, `discord.HTTPException`, any
  generic `Exception` → log `eldritch.budget_dm.send_failed` with class name
  and the first 200 chars of str(exc). NEVER raise.
- Update last-sent timestamp ONLY on successful send (so a 403'd attempt
  still allows the next event-type DM through immediately).
- `attach_to_degraded_mode()` registers a callback that maps
  `degraded_mode_entered` → `notify("degraded_mode_entered", reason)` and
  `degraded_mode_exited` → `notify("degraded_mode_exited", None)`. Store
  the bound callback so `detach` can remove it.

Message format:
- `budget_breached`: `"⚠️ EldritchDM: daily LLM budget breached. Reason: {reason}"`
- `degraded_mode_entered`: `"🛡️ EldritchDM entered degraded mode. Reason: {reason}"`
- `degraded_mode_exited`: `"✅ EldritchDM exited degraded mode."`

Module imports `discord` at top-level (project already depends on it). Use
`from discord import Forbidden, NotFound, HTTPException` defensively wrapped
in a try/except ImportError → assign sentinel classes so unit tests can run
without discord (the project pins discord.py 2.7.1 so this is belt-and-
suspenders).
</behavior>

### Task 6 — Extend `start_schema_version_poller` with `on_schema_change` callback
<task type="auto">
<files>src/eldritch_dm/mcp/cache.py</files>

<behavior>
Change `start_schema_version_poller` signature:

```python
def start_schema_version_poller(
    self,
    client: MCPClient,
    *,
    interval_s: float = 60.0,
    on_schema_change: Callable[[], Awaitable[None]] | None = None,
) -> asyncio.Task[None]:
```

Pass `on_schema_change` to `_poll_schema_version`.

Inside `_poll_schema_version`, in the "version differs" block, AFTER the
MCP wipe completes (the existing `_l1.clear()` + L2 delete), call the
callback if set:

```python
if on_schema_change is not None:
    try:
        await on_schema_change()
    except Exception as exc:  # noqa: BLE001
        self._logger.warning(
            "eldritch.cache.partial_wipe",
            old=last_version,
            new=current,
            mcp_cleared=True,
            secondary_error_type=type(exc).__name__,
            secondary_error=str(exc)[:200],
        )
```

Also wrap the MCP wipe itself so if IT raises, we log `partial_wipe` from
the OTHER side and still attempt the callback:

```python
mcp_wipe_ok = True
try:
    async with self._l1_lock:
        removed += len(self._l1)
        self._l1.clear()
    if self._settings.mcpcache_l2_enabled and self._l2_conn is not None:
        cur = await self._l2_conn.execute("DELETE FROM mcp_cache_entries")
        removed += cur.rowcount if cur.rowcount > 0 else 0
        await self._l2_conn.commit()
except Exception as exc:  # noqa: BLE001
    mcp_wipe_ok = False
    self._logger.warning(
        "eldritch.cache.partial_wipe",
        old=last_version,
        new=current,
        mcp_cleared=False,
        primary_error_type=type(exc).__name__,
        primary_error=str(exc)[:200],
    )

if on_schema_change is not None:
    try:
        await on_schema_change()
    except Exception as exc:  # noqa: BLE001
        if mcp_wipe_ok:
            self._logger.warning(
                "eldritch.cache.partial_wipe",
                old=last_version,
                new=current,
                mcp_cleared=True,
                secondary_error_type=type(exc).__name__,
                secondary_error=str(exc)[:200],
            )
        # if MCP also failed, we already logged partial_wipe above
```

The version tracking (`last_version = current`) STILL advances regardless
— otherwise a transient cache wipe failure would cause an unbounded re-fire
on every subsequent poll.
</behavior>
